    - stop() calls server.close()
    - getJobState(jobId) returns JobState from the Map (returns { status: 'pending' } if not found)
    - W9-04: provider.send(chatId, 'hello') causes getJobState(jobId) === { status: 'done', text: 'hello' }
  </behavior>
  <action>
    Create `workers/gateway/providers/web-widget.ts` implementing the design from RESEARCH.md Pattern 4.

    Key implementation points:
    - `import http from 'node:http'` (use node: prefix for built-in modules)
    - Use `import { CONVERSATION_SOURCES } from '@/lib/conversation-sources.js'` — note .js extension needed for gateway's ESM tsconfig
    - `readonly name = CONVERSATION_SOURCES.WIDGET` — so `provider.name === 'widget'`
    - Two internal Maps: `private jobs = new Map<string, JobState>()` and `private chatIdToJobId = new Map<string, string>()`
    - `start()` only needs to start the HTTP server — no `onMessage` callback needed (WebWidgetProvider calls /api/chat-sync directly, NOT through handleMessage)
    - The internal HTTP server handles:
      - `POST /jobs` — parse JSON body (`{ jobId, chatId, message }`), set `jobs.set(jobId, { status: 'pending' })`, set `chatIdToJobId.set(chatId, jobId)`, immediately respond `{ ok: true }`, then call `APP_URL + '/api/chat-sync'` async:
        ```
        fetch(APP_URL + '/api/chat-sync', {
          method: 'POST',
          headers: { 'Content-Type': 'application/json' },
          body: JSON.stringify({ message, chatId, source: 'widget', userId: null }),
        }).then(async (res) => {
          const data = await res.json();
          // chat-sync response text comes back in the response body
          const text = data.text ?? data.message ?? JSON.stringify(data);
          this.jobs.set(jobId, { status: 'done', text });
          this.chatIdToJobId.delete(chatId);
        }).catch((err) => {
          this.jobs.set(jobId, { status: 'error', error: String(err) });
          this.chatIdToJobId.delete(chatId);
        });
        ```
      - `GET /jobs/:jobId` — extract jobId from URL path, return `JSON.stringify(this.getJobState(jobId))`
      - All other methods/paths — return 404
    - `send()` method is still implemented for Provider interface compliance (the outbound polling loop in gateway/index.ts may call it) — maps chatId to jobId and marks done
    - APP_URL: `const APP_URL = process.env.APP_URL ?? 'http://localhost:3000'`
    - Port: `constructor(private port = parseInt(process.env.WIDGET_INTERNAL_PORT ?? '3001', 10))`
    - Export: `export class WebWidgetProvider implements Provider`

    JobState type:
    ```typescript
    type JobState =
      | { status: 'pending' }
      | { status: 'done'; text: string }
      | { status: 'error'; error: string };
    ```

    Note: The `start(onMessage)` signature from Provider interface must be accepted even though `onMessage` isn't used — `WebWidgetProvider` is the only provider that doesn't need identity linking.
