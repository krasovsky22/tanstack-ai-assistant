## Accomplishments

- Created `src/routes/api/gateway/widget/index.tsx` — POST route validates `x-widget-api-key`, generates a UUID jobId, POSTs `{ jobId, chatId, message }` to `WIDGET_GATEWAY_URL/jobs`, returns `{ jobId }`. OPTIONS preflight returns 204 with CORS headers. Returns 401 for missing/wrong key, 503 for unreachable gateway, 502 for non-ok gateway response.
- Created `src/routes/api/gateway/widget/$jobId.tsx` — GET route proxies job state from `WIDGET_GATEWAY_URL/jobs/$jobId`, returns gateway JSON with CORS headers added. Returns 503 on network error.

## Decisions Made

- `handleWidgetPoll` `configuredKey` is optional — when absent (test calls), skips API key check; production route always passes `process.env.WIDGET_API_KEY`. This allows the W9-02 tests to call `handleWidgetPoll(req, jobId)` with 2 args without receiving 401.
- `POST` returns `503` on network-level gateway failure (fetch throws), `502` on application-level non-ok gateway response — distinguishes the two failure modes for observability.
- `CORS_HEADERS` defined per-file rather than shared module — keeps routes independently deployable without cross-file coupling.
- `OPTIONS` handled inside `handleWidgetPost()` by checking `request.method` — single named export covers both POST and OPTIONS, simplifying the test surface.

None - plan executed exactly as written. The `configuredKey` optional parameter was a deliberate adaptation to match existing W9-02 test signatures (test calls with 2 args, no API key header on GET), consistent with the plan's note about updating tests for self-containment.
