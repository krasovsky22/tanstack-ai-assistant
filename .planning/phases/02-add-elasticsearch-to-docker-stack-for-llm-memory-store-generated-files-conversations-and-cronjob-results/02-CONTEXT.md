# Phase 2: Add Elasticsearch to Docker Stack for LLM Memory - Context

**Gathered:** 2026-03-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Add Elasticsearch as a service to the Docker Compose stack. Index conversations/messages, cronjob results, generated file content, and job records into ES (new data only — no backfill). Expose a `search_memory` tool the LLM can call during chat to retrieve relevant past context via full-text search. Postgres remains the source of truth for all relational data.

</domain>

<decisions>
## Implementation Decisions

### Elasticsearch role
- Secondary search/retrieval index on top of Postgres — Postgres stays as source of truth
- No data migration; ES supplements Postgres, does not replace it for any entity
- New data only — no backfill of existing Postgres records on first run

### Data indexed into Elasticsearch
All four data types get indexed:
- **Conversations + messages** — conversation title, message text parts, source (telegram, browser, cronjob), timestamp
- **Cronjob results** — cronjob name, result text, error, duration, ranAt timestamp
- **Generated files content** — text content of resume/cover letter files generated by the jobs worker
- **Job records** — title, company, description, skills array, status

### LLM memory access
- `search_memory` tool added to the existing tool-calling system (alongside `list_cronjobs`, MCP tools in `src/tools/`)
- LLM decides when to invoke it and what to query — not auto-injected into system prompt
- Search type: full-text keyword search (standard ES `multi_match` or `query_string`), no vector embeddings

### `search_memory` tool response shape
- Returns top N results (5-10) with summaries, not full document content
- Each result includes: `source_type` (conversation | job | cronjob_result | generated_file), short text snippet, timestamp
- LLM reads the results and decides what's relevant to the current conversation

### Claude's Discretion
- ES index names and field mapping design
- Document ID strategy (use Postgres UUID where available)
- Exact `N` for top-N results (5-10 range)
- Write path timing (synchronous dual-write vs. async after Postgres write)
- ES Dockerfile/image version choice
- Tool description and search field list

</decisions>

<specifics>
## Specific Ideas

- The tool fits naturally into the existing architecture: register it in `buildChatOptions()` in `src/services/chat.ts` alongside the cron tools
- Generated file content should be extracted as plain text before indexing (resumes are likely markdown or HTML from the puppeteer PDF flow)
- ES service in docker-compose.yml should follow the same pattern as `postgres`: named volume for persistence, environment vars for config, internal Docker network hostname

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/tools/crontool.ts` + `src/tools/index.ts`: Pattern for defining and registering LLM tools — follow this for `search_memory` tool in `src/tools/memory.ts`
- `src/services/chat.ts` `buildChatOptions()`: Where all tools are registered — add `search_memory` tool factory here
- `src/db/schema.ts` pgTable pattern: Reference for understanding what fields exist on conversations, messages, cronjobLogs, jobs
- `workers/jobs/` generate-resume flow: Entry point for understanding where generated file content can be intercepted for indexing

### Established Patterns
- Tools defined with `toolDefinition({ name, description, inputSchema }).server(async (input) => result)` pattern
- Service modules in `src/services/[name].ts` — ES client and indexing logic should live in `src/services/elasticsearch.ts` or `src/services/memory.ts`
- Console logging prefix pattern: `[memory]` or `[elasticsearch]` for new service logs
- Environment variables loaded via `.env` and documented in `.env.example`

### Integration Points
- `docker-compose.yml`: Add `elasticsearch` service alongside `postgres`
- `src/tools/index.ts`: Export new `memoryTools()` factory function
- `src/services/chat.ts` `buildChatOptions()`: Register memory tools
- Write hooks for indexing:
  - `src/routes/api/conversations/index.tsx` (POST saves conversations) — index after Postgres write
  - `workers/jobs/` generate-resume — index file content after generation
  - Cronjob result write in cron worker — index cronjob log after DB insert
  - `src/routes/api/jobs/index.tsx` (POST creates jobs) — index on creation, update on status change

</code_context>
